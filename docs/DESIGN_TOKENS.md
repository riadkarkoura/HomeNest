# HomeNest — Design Tokens

> Sprint 9.2 — Brand Identity & Design System (planning phase)
> Architecture only. Zero implementation — no CSS variables, no Tailwind
> config, no code, no hex values, no font names, no spacing numbers.
> Source of truth: `docs/BRAND_FOUNDATION.md`. Informed by
> `PROJECT_VISION.md`, `docs/UX_AUDIT.md`, and the existing
> `DESIGN_SYSTEM.md` (which documents HomeNest's current *concrete* values
> — this document defines the *semantic system* those values should
> eventually be organized under, not a replacement for them).

---

# 1. Purpose

A design token is a named, semantic decision — not a value. "The color
used for a destructive action" is a token. Whatever specific color that
token currently resolves to is a separate, later decision. The
distinction is the entire point: code, components, and future pages
should reference the *name*, never the raw
value, so that the meaning of a decision and the value behind it can
change independently of each other.

Design tokens exist to solve three problems that the Sprint 9.0 audit
found HomeNest already living with in practice, even without a formal
token system:

1. **Consistency without central enforcement.** Today, staying consistent
   depends on a contributor reading `DESIGN_SYSTEM.md` and correctly
   recalling or re-deriving the right value for a new situation. A token
   system replaces "remember the right value" with "reference the right
   role" — the value is looked up, not recalled.
2. **A single point of change.** `docs/UX_AUDIT.md` found real instances
   of the same fact (a shipping threshold, a brand's tone of voice) stated
   differently in different places because each instance had been written
   independently rather than resolved from one shared source. Tokens make
   that class of drift structurally harder — a value lives in exactly one
   place and is referenced everywhere it's needed.
3. **A traceable link from brand to pixel.** `docs/BRAND_FOUNDATION.md`
   §16 states that every specific choice in `DESIGN_SYSTEM.md` should be
   explainable by pointing at a brand principle. Tokens are the literal
   mechanism that makes that traceability real: a token's *role* is
   justified by a brand principle, and everything built from that token
   inherits the justification automatically.

This document defines the semantic layer — the roles, the philosophy, the
naming approach — that a future implementation pass will assign concrete
values to. It does not assign those values itself.

---

# 2. Token Philosophy

Five principles govern every token this system will ever define:

- **Semantic over literal.** A token is named for what it *means*
  ("the surface a card sits on") never for what it currently *is* ("light
  gray"). A name that survives its value changing is a correctly-named
  token; a name that would become misleading if the value changed is not.
- **Reusable.** A token must serve many components and contexts across the
  product, not describe a single instance. If a value is only ever needed
  in one place, it is not yet a candidate for tokenization — see §19.
- **Accessible.** Any token that affects what a person can perceive or
  operate (color pairs, touch targets, motion, focus) must have
  accessibility requirements built into its definition, not treated as a
  check performed after the token is chosen. An inaccessible token is a
  broken token, not an acceptable one with a known issue.
- **Theme-ready.** Every token resolves through this semantic layer so
  that a future context (most concretely, dark mode — see §16) can supply
  a different resolved value for the same name without touching anything
  that referenced it. The indirection is what makes future theming
  possible without a rebuild.
- **Platform-independent.** A token describes a design *decision*, not a
  syntax. The same role ("the app's primary accent") should be equally
  meaningful whether it's eventually expressed as a Tailwind class, a CSS
  custom property, or generated by a future tool or agent — consistent
  with `PROJECT_VISION.md`'s long-term AI-native direction, where a
  system that only makes sense to one specific implementation technique
  is a system that will need to be rebuilt sooner than necessary.

A sixth, connecting principle sits above these five: **every token role
must be traceable to a principle in `docs/BRAND_FOUNDATION.md`.** A role
that can't be justified that way isn't a HomeNest token — it's an
implementation detail masquerading as one.

# 3. Color Roles

No colors are chosen here — only the *job* each role does. A future
implementation pass assigns values; this section defines what each value
will be responsible for meaning.

| Role | Purpose |
|---|---|
| **Primary** | The default, most prominent action or brand presence in a given context. Used for the one thing a screen most wants a person to do. Never doubled up — a context has one Primary at a time. |
| **Secondary** | A supporting action that must coexist with a Primary action without competing with it for attention. |
| **Accent** | The brand's single, deliberate highlight. Reserved for moments that must be noticed on purpose — per `docs/BRAND_FOUNDATION.md`'s "one accent, used with intention" — never used as decoration. |
| **Surface** | The background of a distinct, elevated content area (a card, a panel) — what separates a piece of content from the page it sits on. |
| **Background** | The base backdrop of a page or screen, beneath every Surface. |
| **Foreground** | The default color for content (typically text) sitting on a Background or Surface — always paired with the role it's rendered on, never chosen in isolation (see §15 Contrast). |
| **Success** | Confirms a positive outcome or state — an action that completed as intended. |
| **Warning** | Flags something that deserves attention but does not block progress. |
| **Error** | Signals a failure, a problem, or a destructive/irreversible action — the role with the strongest obligation to be unambiguous. |
| **Border** | Delineates or separates regions without introducing a new Surface — the quietest of the roles, used to organize rather than to draw attention. |
| **Muted** | Deliberately deemphasized content — secondary text, disabled states, anything that must visually recede rather than compete. |

This list is a starting set, not a ceiling — new roles may be proposed
under the rules in §18, but every existing use case in the product today
maps onto one of the roles above.

# 4. Typography Tokens

Six roles, each describing *when* it's used, not *how* it looks:

- **Display** — the largest, most editorial text a page can show. Reserved
  for a single defining moment per page (a hero headline) — never used for
  repeated or list-based UI, because its impact depends on scarcity.
- **Heading** — introduces a major section of content. Establishes a clear
  level of the page's hierarchy without attempting to compete with
  Display for attention.
- **Title** — names a specific, individual piece of content within a
  section (a card's subject, an item's name) — one step down from
  Heading, used far more frequently and in denser contexts.
- **Body** — the default role for real, sustained reading content.
  Legibility is the only priority for this role; nothing about Body may
  compromise how comfortably it reads.
- **Caption** — supplementary or secondary information attached to other
  content (metadata, fine print, timestamps). Always subordinate to
  whatever it's captioning; never used for content that stands alone.
- **Label** — short, functional text identifying a control (a button, a
  form field, a tab). Optimized for instant recognition at a glance, a
  different goal from Body's optimization for comfortable reading.

# 5. Spacing Scale

**Philosophy:** spacing is a relationship signal, not a decoration.
Elements placed close together read as related; elements placed apart
read as distinct. A spacing system exists so that "how far apart should
these two things be" always has a small, predictable set of correct
answers — never an arbitrary one chosen fresh for each new layout.

The scale should be built from a small number of base steps that combine
to produce every larger spacing need, rather than maintaining a long list
of unrelated one-off values. A limited, compounding scale is what allows
any two components built independently to still align correctly when
placed together.

**Naming convention:** steps are named by their *relative* position in the
scale (for example, an ordinal or size-word progression such as tightest
→ tighter → standard → looser → loosest), never by their literal value.
A name like "the gap between related inline elements" or "the space
between a section's header and its content" describes intent; a name
like "eight pixels" describes an implementation detail that will need to
be renamed the moment the scale is retuned. No numeric values are defined
in this document — the naming convention only establishes that the scale
is ordinal and relative, not what its steps resolve to.

---

# 6. Radius Tokens

Semantic levels, each tied to how much visual "softness" a shape should
communicate — directly serving `docs/BRAND_FOUNDATION.md`'s "intentional
softness" design principle:

- **Sharp** — no rounding. Reserved for the specific existing elements
  `DESIGN_SYSTEM.md` already identifies as intentionally sharp-edged and
  not to be retrofitted — a preservation role, not a default.
- **Subtle** — the smallest amount of softening, for small interface
  elements (tags, small chips) where rounding should be barely noticeable.
- **Standard** — the default level for most new containers (cards, panels,
  inputs) — the role most new UI should reach for without further
  thought.
- **Pronounced** — a more noticeable curve for larger surfaces or feature
  moments that want to read as distinctly softer than standard content.
- **Full** — fully rounded, for pills, toggles, and circular elements
  (avatars, badges) where the shape itself is the point.

---

# 7. Elevation Tokens

HomeNest's existing shadow guidance is already restrained — soft, minimal,
and often replaced by motion rather than shadow for hover feedback. This
system formalizes that restraint into levels rather than loosening it:

- **Flat** — no elevation. The default for most content, sitting directly
  on the page.
- **Resting** — the smallest possible lift, for elements that need to read
  as subtly distinct from the page around them (a card among other
  content) without appearing to float.
- **Raised** — a clearer separation, for elements that sit above regular
  content but not above everything (a sticky element, a persistent
  panel).
- **Overlay** — the highest level, reserved for content that must command
  full attention above everything else (modals, full-screen overlays) —
  always paired with the Opacity system (§9) for backdrop treatment.

Every level in this system must stay soft and minimal in its eventual
resolved value — adding levels is not license to introduce heavier
shadows than HomeNest currently uses.

---

# 8. Motion Tokens

Four levels, described by the *feeling* they should produce — no timing
values are defined here:

- **Instant** — perceptually immediate. Used for feedback that must feel
  like a direct extension of the person's own action (a press, a toggle),
  where any perceptible delay would feel like the interface didn't
  respond.
- **Fast** — quick and frequent. Used for small transitions a person will
  see often (hover reveals, small state changes) — present enough to feel
  intentional, brief enough to never ask for patience.
- **Normal** — the default pace for content entering or transitioning.
  Long enough to feel considered and calm, short enough to never feel
  slow — the level most motion in the product should use.
- **Emphasized** — reserved for rare, significant moments (a first page
  load, a hero entrance) where a slower, more cinematic pace reinforces
  that the moment matters. Used sparingly on purpose — overuse would
  cheapen exactly the significance this level exists to create.

Every level must define what it collapses to under a reduced-motion
preference (see §15) — this is a required second definition for each
level, not an optional override applied elsewhere.

---

# 9. Opacity Tokens

**Purpose:** opacity is a semantic state tool, not a per-component
aesthetic choice. A small, consistent scale ensures that "this is
disabled" or "this is a translucent overlay" always reads the same way
everywhere it appears, rather than each component inventing its own level
of fade.

Two distinct uses should be named separately rather than sharing one
scale:

- **Content opacity** — dimming a piece of content itself to signal a
  state (disabled, inactive, muted). This use answers "how much has this
  element been deemphasized?"
- **Layering opacity** — the translucency of a surface stacked over other
  content (a backdrop behind an overlay, a glass-effect panel). This use
  answers "how much of what's underneath should still be visible?"

Conflating the two would let a state-driven dimming decision accidentally
double as a layering decision, or vice versa — they should remain
independently named even though both are expressed as opacity.

---

# 10. Border Tokens

A semantic hierarchy of emphasis, always paired with a color role from §3
(a border token defines *how strongly* a border should read; the Border
color role defines *what it's rendered in*):

- **None** — no border. Separation is communicated by spacing or surface
  contrast alone.
- **Subtle** — the default, low-emphasis separator (between list items,
  between minor regions).
- **Standard** — the default border for containers and inputs that need a
  clear, but still restrained, edge.
- **Emphasis** — a stronger border reserved for states that must draw
  clear attention (focus, active selection, error) — the border-system
  counterpart to the Error and Accent color roles.

---

# 11. Icon Tokens

Icons are a supporting vocabulary, not an independent design surface:

- Every icon resolves to a semantic **size** role, scaled to sit correctly
  beside the typography token it accompanies — an icon should never look
  mismatched against its adjacent label.
- Every icon resolves to a semantic **color** role from §3, never a
  one-off value chosen for that single icon.
- The icon vocabulary draws from a single, sanctioned icon set (as
  `DESIGN_SYSTEM.md` already establishes) so the visual language stays
  singular across the product — introducing a second icon source is a
  brand-consistency risk, not a neutral choice.
- Any icon used without accompanying visible text must resolve to an
  accessible label — icon tokens are incomplete without this pairing (see
  §15).

---

# 12. Grid Tokens

Layout philosophy, not layout specification:

- Content should resolve to a small, fixed set of standard container
  widths, so unrelated pages still feel like they belong to the same
  product rather than each page inventing its own maximum width.
- Column and gap behavior should scale predictably across breakpoints as
  a system, not be redesigned per page — a grid token answers "how does
  this layout adapt," once, for every page that uses it.
- The purpose of grid tokens is to prevent every new page from re-deriving
  its own layout math from first principles — consistency of *structure*,
  not just of individual values.

---

# 13. Breakpoint Strategy

**Strategy: mobile-first.**

**Reasoning:**
- `docs/BRAND_FOUNDATION.md` §8 states plainly that mobile is the default
  experience, not a fallback — a desktop-first approach treats mobile as
  a constraint to accommodate after the fact, which is the opposite of
  that principle.
- Starting from the most constrained viewport forces every essential-
  content decision to happen first, before there's room to hide
  complexity behind extra desktop space. `docs/UX_AUDIT.md` found this
  discipline already holding up well in practice across the product's
  history — mobile-first has been the working approach in fact, even
  before being formalized as architecture here.
- It is cheaper and safer to progressively *enhance* a design for
  additional space than to progressively *strip down* a design that
  assumed space it won't have.

Desktop-first is explicitly rejected as this system's default strategy —
any exception would need to be justified per §18's governance rules, not
adopted casually per component.

---

# 14. Responsive Principles

- **No feature is exclusive to one breakpoint.** Anything that works on a
  larger viewport must have a working equivalent on a smaller one —
  never present, then silently absent.
- **Breakpoints change layout, not functionality.** A capability available
  at one width must remain available at another; only its presentation
  may adapt.
- **Hover and touch are equally first-class.** Any interaction revealed
  by hover must have an equivalent path for touch — hover can enhance an
  experience, but it may never be the *only* way to reach something.
- **Scale fluidly where possible, jump only where necessary.** Typography
  and spacing should prefer smooth, continuous scaling across viewport
  sizes over abrupt jumps at fixed breakpoints, consistent with how
  `DESIGN_SYSTEM.md`'s existing fluid type scale already behaves — that
  behavior is elevated here from a technique into a standing principle.

---

# 15. Accessibility Tokens

- **Focus states** — every interactive element resolves to one consistent,
  clearly visible focus token. A focus indicator may never be suppressed
  without an equally visible replacement defined at the same time.
- **Contrast** — Foreground/Background and Foreground/Surface combinations
  are defined as validated *pairs*, not independent tokens combined ad
  hoc. Pairing two color roles must never be able to silently produce an
  insufficient-contrast result — compliance is a property of the token
  system itself, not a manual check performed after the fact per
  instance.
- **Touch targets** — every interactive element resolves to a minimum
  tappable-area token regardless of its visual size, so a small icon can
  still carry an adequately large touch target.
- **Reduced motion** — every Motion token (§8) must define its reduced-
  motion equivalent as part of its own definition, not as a separate,
  optional override system layered on afterward.

---

# 16. Dark Mode Strategy

High-level strategy only: because all color usage in this system flows
through the semantic Color Roles defined in §3, dark mode is a strategy
of resolving those same role *names* to a different set of values for a
dark context — never a parallel, separately maintained set of dark-
specific components or one-off overrides scattered through the codebase.

The *meaning* of "Primary," "Error," "Surface," and every other role must
stay identical between light and dark contexts; only the value each role
resolves to changes. This is what the "theme-ready" principle in §2
exists to guarantee, and it's the only reason dark mode is a future
option rather than a future rebuild.

Dark mode is explicitly out of scope for this planning document and is
not being scheduled by defining this strategy — this section only
establishes that the token architecture above is what makes adding it
later possible without restructuring everything that came before it.

---

# 17. Naming Convention

Future tokens should be named by **category, then role, then modifier**
— a structure where the name alone communicates a token's family and
intent without requiring a lookup of its value (for example: a spacing
token's name should say it's a spacing token, in which relative step of
the scale, in that order — never encoding the literal resolved value into
the name itself).

Two hard rules follow from this structure:

- A token's name must never describe its current value (never name
  something after a color it currently resolves to, or a pixel amount it
  currently equals) — that couples the name to a fact that's expected to
  change, defeating the purpose of tokenizing it at all.
- A token's name must remain stable across theme or context changes — the
  same name should mean the same *thing* in light mode and dark mode, on
  mobile and desktop, today and after a future rebrand of the specific
  values behind it.

---

# 18. Token Governance

Rules for adding to or changing this system:

- **A new token may be added only if no existing token can reasonably
  serve the same semantic role.** The default assumption is reuse; a new
  token is the exception that must be justified, not the default.
- **Every new token must trace to a principle in
  `docs/BRAND_FOUNDATION.md`.** If a proposed token can't be justified by
  an existing brand principle, that's a signal the token — or the
  principle — isn't ready yet.
- **Changing a token's resolved value must never require changing its
  name**, or any code that references it. If a value change forces a
  rename everywhere it's used, the token was named after its value
  instead of its role, and should be corrected under §17 rather than
  repeatedly patched.
- **New tokens are proposed and reviewed, not added ad hoc inside a
  single component.** This mirrors the ADR discipline already used
  elsewhere in this project (`docs/DECISIONS.md`) — a token is a shared,
  architectural decision, not a local implementation choice.
- **A component-specific one-off value is never an acceptable substitute
  for extending the token system properly** — see §19.

### Token Stability

Once a token's semantic meaning is established and adopted by components,
that meaning must remain stable. If a new semantic role is needed later,
introduce a new token rather than repurposing an existing one — this is
what protects long-term consistency and prevents regressions across
everything already built on top of it.

---

# 19. Anti-patterns

Things that must never happen under this system:

- **Magic values.** Any color, spacing, radius, or timing chosen directly
  in a component without going through a named token — even once, even
  for "just this one case." A value that isn't a token is untracked,
  unreusable, and invisible to this entire system.
- **Hardcoded colors.** A literal color value referenced anywhere outside
  the layer that defines what each Color Role resolves to.
- **Random spacing.** A gap, margin, or padding value chosen because it
  "looked right" in one place, rather than selected from the spacing
  scale defined in §5.
- **Component-specific tokens.** A token whose name or definition only
  makes sense for one specific component (for example, naming something
  after the component it was built for rather than the role it plays) —
  this defeats the "reusable" principle in §2 and produces a token system
  that grows without bound.
- **Inventing a new token when an existing role already fits.** Choosing
  novelty over reuse, in violation of §18's governance default.
- **A Motion token defined without its reduced-motion equivalent**, or an
  Accessibility Token (§15) treated as optional for a specific instance
  rather than a floor that always applies.
- **A dark-mode-specific override introduced outside the semantic Color
  Role system** — any dark-mode value that isn't just a different
  resolution of an existing role (§16) is an architecture violation, not
  a shortcut.

---

# 20. Definition of Done

This document is complete when the token *roles, philosophy, naming
convention, and governance rules* are fully defined — and contains **zero
implementation**: no CSS variables, no Tailwind configuration, no code of
any kind, no HEX or other color values, no font names, and no spacing,
radius, timing, or opacity numbers. Everything above is architecture and
system design only.

This document succeeds if a future implementation sprint can assign
concrete values to every role defined here — colors to §3, sizes to §4,
a numeric scale to §5, and so on — without needing to invent a single new
role first. If a future implementer finds themselves needing a role this
document doesn't already provide, that's this document's gap to close
under §18, not a reason to introduce an untracked one-off value under
§19.

This document does **not** define: any actual value, any code, any
component, or any visual redesign — those remain entirely out of scope
for Sprint 9.2's planning phase, per explicit instruction.
